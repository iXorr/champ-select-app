# Информационная система «Кофеманы»

## Замечания

1. Функция для CSV-импорта НЕ БЕРЁТ в расчёт названия колонок: только их порядок (и для импорта товаров нужно добавить, также, возможность добавить ДОП. ПОЛЯ - roast_level и country).
2. Я не знаю, почему ``orders.created_at`` в DashboardController и OrderController работает.
3. Софт-делит для заказов не нужен: достаточно просто менять статус на "Отменён".

## Описание задания

Необходимо разработать информационную систему для компании **«Кофеманы»**, занимающейся продажей кофе и сопутствующей продукции. Цель системы — автоматизация процессов продаж и ведение учета с использованием удобного и понятного интерфейса.

Вся информация системы должна храниться в базе данных. При проектировании базы данных необходимо строго придерживаться **правил нормализации (НФ)**.

Система должна быть удобной для использования как на компьютере, так и на планшете с разрешением **1024×768 px**, включая адаптивную верстку и микроанимации для улучшения пользовательского опыта.

Материалы для выполнения задания (CSV-файлы) предоставлены на сервере.

## Страница аккаунта

Позволяет пользователю просматривать информацию о своем аккаунте:

* ФИО
* Email
* Логин
* Тип аккаунта

### Смена пароля

* Требуется ввод текущего пароля
* Ввод нового пароля с подтверждением
* Если текущий пароль введен неверно, выводится сообщение:

> **«Пароль неизменен, вы ввели неверный текущий пароль»**

## Раздел «Товары» (CRUD, только администратор)

### Основные поля товара (обязательные):

* Наименование товара
* Производитель
* Фотография:

  * Максимум 2 МБ
  * Форматы: JPG, JPEG
  * Если не выбрана — используется `Заглушка.png`
* Стоимость
* Единица измерения: кг, шт, г, л, мл
* Краткое описание (до 255 символов)
* Полное описание

### Дополнительные характеристики

* Хранятся отдельно
* Выбираются и заполняются для каждого товара индивидуально

### Ограничения

* При редактировании запрещено менять:

  * Наименование товара
  * Единицу измерения
* Удаление товара возможно **только если он не использовался в заказах**

### Импорт товаров

* Импорт из CSV-файла `товары.csv`
* Загружаются все основные поля, **кроме фотографии**

## Раздел «Клиенты» (CRUD)

### Поля клиента (обязательные, если не указано иное):

* ФИО

  * Проверка на минимум 2 слова (отчество необязательно)
* Email

* Адрес доставки
* Контактный номер
* Пометка (необязательное поле)

### Ограничения

* Удаление клиента возможно **только если он не использовался в заказах**

### Импорт клиентов

* Импорт из CSV-файла `Клиенты.csv`
* Обязательны все поля, кроме «Пометка»

## Раздел «Заказы» (CRUD)

### Общий список заказов

Отображаемые данные:

1. Номер заказа
2. Клиент
3. Пользователь, создавший заказ
4. Общая сумма
5. Статус заказа
6. Дата создания
7. Дата отгрузки (если отсутствует — отображается `—`)
8. Кнопки изменения статуса

### Статусы заказа

* Новый
* Отгрузка
* Доставка
* Выдан
* Отменен

**Правила изменения статусов:**

* Нельзя перейти к предыдущему статусу
* При смене статуса с «Новый» на любой, кроме «Отменен», если дата отгрузки не заполнена — она устанавливается автоматически

---

### Создание и редактирование заказа

Содержит:

* Клиент
* Дата заказа (автоматически при создании)
* Дата отгрузки (при статусе «Отгрузка»)
* Статус заказа
* Позиции товаров (загружаются через API):

  * Наименование
  * Фотография
  * Единица измерения
  * Стоимость
  * Скидка (по умолчанию 0)
  * Количество (минимум 1)
  * Сумма

**Формула расчета суммы позиции:**

```
(Стоимость − (Стоимость / 100 × Скидка)) × Количество
```

### Удаление заказа

* Фактическое удаление не выполняется
* Удаление означает перевод заказа в статус **«Отменен»**

## Фильтрация заказов

Должна поддерживаться фильтрация по:

* Статусу

* Дате отгрузки

Фильтры должны работать:

* По отдельности
* В совокупности с использованием логики **AND**
